# 카파시 LLM Knowledge Base 탐험 일지

## 핵심 요약
> 카파시가 제안한 "RAG 대신 LLM이 직접 저작하는 마크다운 위키" 아이디어를 내 상황에 어떻게 적용할지 5시간 탐구. 결론: 새 인프라 만들지 말고 기존 PARA 인박스부터 시작. 자동화는 손으로 3~5개 써본 뒤.

## 오늘 뭘 알게 됐나

### 시작: karpathy/autoresearch
- 카파시가 만든 "AI 에이전트가 밤새 nanochat 학습 실험 자동으로 돌리는" 레포
- 5분 고정 시간 예산 + `val_bpb` 메트릭 + 에이전트가 `train.py` 만 수정
- **H100 1장 필요** → 나한텐 안 맞음
- 근데 철학은 꽂힘: "에이전트에게 경계 주고 그 안에서 자율 반복"

### 진짜 관심 주제: LLM Knowledge Base
- 며칠 뒤 카파시가 X에 올린 후속 아이디어
- **"RAG 집어치우고 LLM이 직접 관리하는 마크다운 위키를 만들어라"**
- `raw/` → LLM 사서 → `wiki/` → Obsidian 뷰
- 벡터 DB 없음, 검색 서비스 없음, 그냥 파일 + agentic file navigation
- Health check = "지식베이스에 린트 돌린다"
- 규모: 100 아티클 / 40만 단어. 그 이상은 카파시도 테스트 안 했음

## 왜 이게 RAG보다 나은가 (카파시 주장)

1. **블랙박스 없음** — 사람이 읽을 수 있는 마크다운
2. **retrieval noise 없음** — 구조화된 위키 + 백링크
3. **유지보수 간단** — 파일 편집만
4. **확장성 함정 없음** — mid-size엔 RAG가 과함
5. **static snapshot 아님** — LLM이 계속 저작

## 내가 떠올린 중요한 직감들

### 1. 개인 vault ↔ AI 관리 vault 분리
- 개인 노트는 "정리 안 된 게 가치" (fleeting notes)
- AI 위키는 "정리된 게 가치"
- 섞으면 둘 다 망함
- → 처음부터 분리해야 함

### 2. ClickHouse는 wiki의 "집"이 아니라 "원료"
- 동료가 "CH에 wiki 넣자" 얘기했음
- 근데 CH에 마크다운 넣으면 카파시 방식의 모든 장점이 사라짐 (Obsidian, git, 백링크, 투명성)
- CH는 raw 이벤트(릴리즈, Sentry, 배포)가 살 자리
- Wiki는 그걸 **읽어서** 파생된 큐레이션 층
- → 두 레이어 분리가 핵심

### 3. 스케일 걱정은 두 개로 나눠야 함
- ❌ Wiki 자체가 커져서 LLM이 못 찾을 걱정 — **안 커지게 유지** (100-300 아티클 상한)
- ⭐ 코드베이스가 방대해서 wiki로 다 못 커버할 걱정 — **진짜 문제**
- 답: **Wiki는 코드를 복제하지 않는다**. "왜/언제/주의" 만 다루고, "무엇/어떻게"는 코드가 진실. Claude가 필요하면 코드 직접 읽음
- Pain-driven growth: 아픔이 일어날 때만 아티클 추가

### 4. 스킬 vs 콘텐츠는 다른 것
- 스킬(도구) = claude-toolkit에 감
- 위키(콘텐츠) = second-brain에 감
- **콘텐츠 없이 스킬 만들면 쓸모없음** — "좋은 아티클"의 스펙이 없어서
- 순서: 1) 손으로 3~5개 쓰기 → 2) 패턴 발견 → 3) 그제서야 스킬 자동화

## 오늘 가장 중요한 깨달음

> **"자동화보다 수동이 먼저다. 좋은 아티클이 뭔지 모르면 자동화는 쓸모없는 것을 대량 생산할 뿐이다."**

처음엔 "레포 긁고 슬랙 읽어서 자동으로 위키 생성"이 매력적이었는데, 그거부터 시작하면 100% 망함. 손으로 3~5개 쓰면서 감 잡아야 스킬이든 MCP든 만들 수 있음.

- "AI 에이전트가 회사 맥락 모른 채로 매번 시작" = 진짜 아픔
- 이걸 풀 수 있으면 커리어 자산으로 엄청남
- 근데 **지금 당장은 아님**. 개인 OSS로 검증 먼저, 그 다음 회사 제안
