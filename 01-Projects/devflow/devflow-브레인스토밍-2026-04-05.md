# devflow 브레인스토밍 — 2026-04-05

> 하루 종일 devflow(https://github.com/lucid-jin/devflow) 제품 방향 고민한 원본 기록.
> 결론/요약이 아니라 **고민 과정 전체**를 남김. 나중에 같은 함정 안 밟으려고.

## 핵심 요약

> **"Linear 쓰는 개발자용 cmux"로 포지셔닝. 하지만 시장성은 낮음 — ★100~300급 개인 OSS 각오. 본인이 매일 쓰는 도구로 만들기.**

## 📌 오늘 내린 결정

1. **정체성 B 선택**: "Linear 쓰는 개발자용 터미널 네이티브 TUI 세션 매니저". 초기 비전(A. 자동화 도구)은 이미 완성됐고, C(워크벤치)는 야심 있지만 B 없이 가면 기초 부실
2. **cmux 기반이 아닌 Ghostty 기반**: cmux는 GUI, devflow는 터미널 네이티브. 트리거는 cmux/터미널에서 수동 1회 호출 → 알아서 세션 경로 선택
3. **트래커 agnostic 방향**: 꼭 Linear 아니어도 됨. 비개발자에게 특정 페이지 찾아가는 게 어려운 문제도 해결할 수 있음
4. **worktree 격리 추가 — 1+2+6만**: node_modules 독립 설치 + `.env` 자동 복사 + 빌드 캐시 분리. 포트/DB/FS 샌드박스는 굳이
5. **MVP = 기존 기능 완성 먼저**: 새 기능 짓기 전에 "된 부분" 완성
6. **시장성은 중요하지 않음**: 본인이 매일 쓰는 도구로 만들기

## 🧭 제품 방향 도식화

### 오늘 느낀 긴장

```
 [초기 devflow]                    [오늘 논의한 devflow]
      ↓                                    ↓
"Linear 이슈 하나 →               "터미널 네이티브 AI 세션 매니저.
 worktree + Claude 세션             여러 세션을 관리/조율/적립하는
 자동 세팅"                          플랫폼."

 좁고 깊음                           넓고 얕음
 시작의 마찰 0                       작업 전주기 관리
 경쟁자: 수동 셋업                   경쟁자: cmux, agf
 완성 기준: 명령 한 번이면 끝        완성 기준: 매일 켜는 대시보드
```

둘 다 의미 있는 제품인데 **다른 제품**. 동시에 만들려 하면 정리가 안 됨.

### 세 가지 가능한 정체성 (A/B/C)

```
  스코프 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━→
  좁음                                      넓음

  ┌─────────┐
  │    A    │  워크플로우 자동화 (초기 비전, 이미 완성)
  │  Launch │  "Linear → Claude까지 한 번에"
  └─────────┘
       ↓
  ┌──────────────────┐
  │        B         │  세션 매니저 (오늘 선택)
  │ Launch + Manage  │  "Linear 쓰는 개발자용 cmux"
  └──────────────────┘
       ↓
  ┌────────────────────────────────┐
  │              C                 │  AI 코딩 워크벤치 (v0.3+)
  │ Launch + Manage + Close Loop   │  "작업 전 주기를 Linear 중심으로"
  └────────────────────────────────┘
```

| | A. 자동화 도구 | B. 세션 매니저 | C. 워크벤치 |
|---|---|---|---|
| **한 줄 설명** | Linear 이슈 하나를 즉시 Claude 세션으로 | Linear 쓰는 개발자용 cmux | AI 코딩의 시작~종료 전 주기를 Linear 중심으로 |
| **핵심 동사** | Launch | Launch + Manage | Launch + Manage + Close |
| **킬러 기능** | `devflow` → 이슈 고르면 끝 | Sessions 탭, attach | `devflow done` + changeset + Linear 연동 |
| **MVP 크기** | 이미 거의 완성 | +Sessions 탭 +attach | +done +적립 +대시보드 |
| **경쟁자** | 수동 셋업 스크립트 | cmux, agf, tmuxinator | Graphite, Linear 자체 |
| **README 한 줄** | "Zero-friction Linear → Claude" | "Terminal-native session manager for Linear devs" | "AI coding workbench built around Linear" |
| **위험** | 너무 작아서 스킬/스니펫 취급 | cmux와 비교돼서 GUI 부재가 약점 | 스코프 폭발, 영원히 미완성 |

## 🎯 포지셔닝 (최종)

**"Linear 쓰는 개발자용 cmux — 터미널 네이티브 TUI 세션 매니저"**

| | cmux | devflow |
|---|---|---|
| 형태 | macOS GUI 앱 (Ghostty) | TUI (SSH/리눅스/tmux 안에서도) |
| 이슈 트래커 | 없음 | Linear 1급 통합 (트래커 agnostic 지향) |
| worktree | 수동 | 자동 생성/관리 + 격리 확장 |
| 세션 시작 | 빈 탭 | 이슈 선택 → 컨텍스트 주입된 Claude 세션 |
| 배포 | brew cask (macOS only) | `go install` (크로스 플랫폼) |

### 해석
- **이 판은 뜨거움**: 최근 1년간 ★1k+ 여러 개
- **GUI가 스타 흡수**: cmux 12k, Crystal 3k — 터미널 TUI/CLI는 같은 기능이라도 훨씬 낮은 tier
- **가장 비슷한 참조 = agf**: 한국 개발자, Rust TUI, 2달 ★107. devflow 기대 궤적도 이 정도
- **Linear 종속은 TAM을 agf보다 더 좁힘** (agf는 7개 에이전트 지원)
- **결론**: ★100~300급 개인 OSS로 남을 확률 높음. 본인이 매일 쓰는 도구로 만드는 게 정답

### 잠재적 성장 각도
agf(read-only) → devflow(write + 트래커 연동)는 agf 사용자의 "다음 단계"로 포지션 가능. agf 생태계에 얻어타는 게 가장 빠른 경로.

## 🔍 현재 구현 감사

> 실제 코드 감사 결과. README와 다른 부분 포함.

### ✅ 실제로 되는 것
- **R1~R5, R7~R13**: Linear OAuth, 이슈 브라우징, worktree 생성, 컨텍스트 주입(`.claude/TASK-xxx.md`), tmux + Claude 부팅, `--no-tmux` 폴백, `devflow init`, Bubble Tea TUI, 필터/레포 전환, TOML 설정 — **모두 동작**
- **세션 registry**: `~/.devflow/sessions.json`에 모든 세션 메타데이터 **저장은 됨**
- **테스트**: config / linear / engine(registry, claude) — 좁지만 존재

### 🐛 유령 기능 (README에 있는데 안 됨)
- **R6. Linear 상태 → In Progress 자동 전환**
  - `linear/client.go`에 `UpdateIssueState()` 메서드는 구현돼 있음
  - 근데 **아무도 호출 안 함**. TUI에서 "Updating Linear status" 단계는 보이는데 실제로는 noop
  - → **1~2줄 수정으로 해결 가능**

### 🕳️ 껍데기만 있는 것
- **Sessions 탭** — `tui/app.go`에 탭 선언만, `View()`가 placeholder 문자열만 반환
- **Dashboard 탭** — 동일
- **세션 registry 조회** — `.List()` 메서드는 있는데 CLI에서 호출하는 곳이 없음. **write-only**

### ❌ 아예 없는 것
- CLI 확장: `devflow list`, `devflow attach`, `devflow done`, `devflow rename`
- 세션 rename (#1 요구사항)
- 트래커 추상화 (Linear가 Engine에 직접 박혀있음)
- AI 에이전트 추상화 (Claude 하드코딩)
- worktree 격리 확장 (node_modules, .env, 캐시)
- changeset 적립
- 병렬 조율 / 과거 세션 검색

### 🎯 MVP 완성 = 4개 작업

| # | 작업 | 왜 쉬운가 | 왜 중요한가 |
|---|---|---|---|
| **1** | R6 버그 픽스: `StartSession`에서 `UpdateIssueState()` 호출 | 메서드 존재, 1~2줄 | README 약속 지키기 |
| **2** | `devflow list` CLI | registry JSON 이미 저장 중, 읽기만 | 세션 파악 가능 |
| **3** | `devflow attach` CLI | registry에 tmux 이름 있음, `tmux attach -t` 한 줄 | 매일 쓰는 기능 |
| **4** | Sessions 탭 실제 구현 | 2·3 로직 재사용 + Bubble Tea 리스트 | Dashboard보다 먼저 필요 |

### 🔁 #1 요구(rename)는 4번 이후 자동 해결
- Sessions 탭에서 `r` 키 → 표시명 변경 → registry에 `display_name` 필드 추가
- Linear 세션이면 이슈 제목을 자동으로 표시명
- 선택적으로 `tmux rename-session` 호출

### 기타 눈에 띄는 것
- `LinearIssueID` 필드 저장되지만 **사용 안 됨** → R6 유령 기능과 연결
- `rand.Read()` 에러 무시 (`linear/oauth.go:38, 48`) → PKCE 약화 리스크
- HTTP 서버 `Shutdown` 누락 경로 → goroutine 누수 가능
- 에러 래핑 일관성 없음 (`%w` vs `%v`)
- 테스트가 happy path만 커버
- `SendKeys("claude")` 실패 시 사용자 모름 (tmux 소켓 권한 등)
- Linear 쿼리 타임아웃 10초 하드코딩, 재시도/사용자 메시지 없음

## 💡 주요 통찰 (다시 안 까먹게)

1. **원래 비전이 무엇이었나 — cmux에서 devflow 호출 1회 수동. 세션 경로 자동 선택. changeset을 Linear 이슈로 대체**
2. **Linear 이슈 = changeset**: 세션이 처음부터 Linear 이슈에 묶이면 별도 changeset 파일 필요 없음. Linear가 변경 적립 저장소
3. **agf와는 상보적, 경쟁 아님**: agf=read-only 세션 파인더, devflow=write 세션 생성기 + 트래커 연동
4. **Sessions 탭을 devflow 안에 또 만들면 cmux와 중복** — 신중히
5. **한국어 사용자 본인이 매일 체감할 버그 있음** (P1 slugify, P2 truncate) — 본인용으로라도 먼저 고치면 즉각 즐거움
6. **registry는 write-only 상태**. `.List()`는 구현됐는데 CLI에서 호출 안 함 → list/attach 구현은 저장소 추가 없이 "읽기만" 하면 됨
